Supported Asset Types

Asset Type Example
IP 127.0.0.1
IPRange 10.0.0.0/16
Hostname www.adevinta.com
DomainName (it has a SOA record) adevinta.com
WebAddress https://www.adevinta.com
AWSAccount arn:aws:iam::111111111111:root
DockerImage registry.hub.docker.com/organisation/image:latest
GitRepository https://github.com/organisation/repository.git
GCPProject gcp-project-123

Asset types are automatically detected when you add an asset using the UI. If you want to force a specific asset type you should either add the asset type after the asset separated by a comma (i.e. arn:aws:iam::111111111111:root,AWSAccount) or use the API. However, please note that the asset type GCPProject is an exception to this auto-discovery and required to added manually. Generally, we recommend you to use the auto-detection. Why?

For example when you add an asset that matches a WebAddress, at least two are created: one WebAddress and one Hostname. In some cases a DomainName asset is created too. For example if you add https://adevinta.com/, the assets created are:

  • WebAddress → https://adevinta.com/
  • Hostname → adevinta.com
  • DomainName → adevinta.com

How am I scanned?

When you add a new asset it is automatically added to the Default group. Using the API you can move it to other groups.

Every month the assets you have in the Default and the Sensitive groups are scanned automatically. For every scanned asset, the appropriate checks are executed depending on its type.

There are some special cases that you should consider:

  • IPRanges are not scanned by default now. If you want them to be scanned contact the Vulcan crew.
  • In order to be able to scan AWS accounts, the Security Audit Role needs to be created in the account.
